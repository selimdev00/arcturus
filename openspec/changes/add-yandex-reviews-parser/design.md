# Design

## Stack

- **Backend**: Laravel 11, PHP 8.3. Sanctum for SPA cookie auth. SQLite (file)
  to keep the prototype single-container and avoid an extra DB service on a
  RAM-tight host; migrations still define a clear schema as required.
- **Queue**: `database` driver + one worker. No Redis container — keeps the
  footprint minimal; the queue is only used for the long-running full parse.
- **Headless**: Spatie Browsershot (drives Puppeteer + Chromium from PHP).
  Chromium ships inside the app image.
- **Frontend**: Vue 3 + Composition API, Vite, built to static assets served by
  the same container.
- **Container**: FrankenPHP single image (PHP + web server + worker), capped at
  ~800 MB memory so a headless run cannot OOM the shared host.

## Parsing strategy (graded core)

Two tiers, both targeting `https://yandex.com` (`.ru` blocks datacenter IPs):

### Tier 1 — fast path (no browser)
`GET /maps/org/<id>/reviews/` server-renders, in the initial HTML state blob:
average rating, ratings count, reviews count, and the first 50 reviews.
`YandexReviewsClient` fetches this with a realistic UA + `Accept-Language: ru`,
extracts the embedded JSON state, returns counters + first page. Used to answer
the settings "save + preview" instantly and to seed the cache.

### Tier 2 — full set (headless, JSON interception)
`YandexReviewsScraper` (Browsershot) opens the reviews card, installs a `fetch`
hook in page context, then scrolls `.scroll__container` to `scrollHeight`
repeatedly until the review count stops growing (Yandex caps at ~600). Each
`fetchReviews` response is captured as clean JSON — we do **not** parse DOM
classes. Yandex's own JS computes the signed `s` param, so we never reverse it.
Runs as a queued job; persists every review.

### Why not call the JSON API directly
`fetchReviews` requires a per-page `s` signature generated by obfuscated client
JS; cold requests return 400 (verified). Driving the browser lets Yandex sign
the requests for us while we keep the clean JSON payload — best of both.

### Caching
Parse once → store `organization` (avg, ratingsCount, reviewsCount, parsedAt)
and `reviews` (reviewId unique, author, text, rating, reviewedAt). The SPA
paginates 50/page from the DB. Re-parse is explicit (button / stale TTL), never
per page view.

## Error handling (graded)

`ParseStatus` enum: `ok | unreachable | markup_changed | empty | captcha`.
- Tier 1 non-200 / empty body → `unreachable`.
- State blob missing expected keys → `markup_changed`.
- Captcha markers in response (`smartcaptcha`, `showcaptcha`) → `captcha`,
  surfaced to UI with a retry affordance; optional residential-proxy fallback
  documented as the production mitigation.
- All wrapped in a service layer; controllers stay thin (TZ requirement).

## URL validation

Accept `yandex.{ru,com,com.tr,…}/maps/org/<slug>/<id>/` and short share links
(resolved via redirect). Extract the numeric `businessId`. Reject anything else
with a translated error.

## Deploy (isolated)

- `/srv/arcturus` on host `snapfrom-prod`, own `docker-compose.yml`, host port
  3800, `mem_limit`.
- Caddy vhost `arcturus.selim.services → localhost:3800`.
- Cloudflare DNS record for the subdomain (proxied).
- No existing stack touched; teardown = remove dir + vhost + DNS.

## Out of scope

- Multi-org management, multi-user, auto-refresh scheduling, review replies UI,
  exceeding Yandex's ~600 cap via multi-ranking merge (documented as "given more
  time" in README).
